Introduction

The {{ site.product_short }} Web Store is a configuration, features and API integration that enables {{ site.product_short }} licensees to implement subscriptions, in Bring Your Own (BYO) web stores. The content owners implement the store front and business logic for renting and selling and the payment system integration in the web store. An {{ site.product_short }} API provides the interface for the store to add, remove and edit the resulting entitlement changes within the {{ site.product_short }} CMS. The logic defining access and display of the content titles in {{ site.product_short }} apps is unchanged.

Implementation of all financial reporting, dashboards, and tax payment are not managed or tracked in {{ site.product_short }} and are the responsibility of the Content Owner.

{{ site.product_short }} Store components

All {{ site.product_short }} Store purchases are made through web stores. The selling model requires Viewers to visit the customer web store, purchase content and then open {{ site.product_short }} apps on their supported devices as a separate action.

The required components are:

{{ site.product_short }} Store Workflows and Configurations

You may well have already had a commerce web sites as an integral component of their businesses prior to introducing {{ site.product_short }} and want to add {{ site.product_short }} subscriptions to your web store products. The web stores consist of three components:

  1. Web store - The web store is your organization’s commerce site. It presents the products, pricing and purchase models.
  2. Payment processor - The payment processor is a system that processes transactions, accepting payment from credit cards, debit cards and e-wallets, deposits the payments in the organization’s bank account and sends receipts to the purchaser.
  3. User Management System - Your organization maintains a list of their registered patrons in their UMS.

The {{ site.product_short }} Store integration has two development activities

  1. {{ site.product_short }} receives notifications of financial transactions from the web store - Sales and cancellations of {{ site.product_short }} subscriptions from the web store are reported to a {{ site.product_short }} API. {{ site.product_short }} adds or removes entitlements based on these notifications.
  2. Connect and synchronize the {{ site.product_short }} and Web Store User Management Systems - The Webstore UMS user record contains the master record and the user and is the source of stored user data. Proxy users are created in the {{ site.product_short }} UMS for each of the registered patrons. {{ site.product_short }} assigns entitlements to the proxy user record upon receiving transaction notifications. Changes in the store UMS such as adding and deleting users are applied to matching user records in the {{ site.product_short }} UMS.

{{ site.product_short }} Store Configurations

{{ site.product_short }} Stores are pre-existing, customer owned and operated BYO web stores that are integrated with {{ site.product_short }} for the purpose of adding {{ site.product_short }} subscriptions to the list of products sold on the web store. Arts organizations provide an example illustrating the two primary selling models for supporting this objective.

Arts organizations currently sell subscriptions, tickets and physical merchandise related to performances through their web stores. Therefore, there are existing web stores, user management, payment processing and entitlement tracking systems in place to manage that commerce.

Some organizations have decided to expand their audience by including virtual access to these same events as well as recorded content. They further want to consolidate the entire commerce management process within their web stores.

A subset of organizations want to achieve the same result with their web stores, and want to sell those same products as in app purchases in the app stores (Apple Store, Google Play, Amazon, Roku and the Stripe integration with the {{ site.product_short }} Web app).

Web Store Only Purchases

Web store only workflow is implemented as shown in Figure 1. The customer brings their own web store, payment processor and UMS. The integration points are:

  • UMS Proxy Users - The full records of the registered users are maintained in the customer’s BYO UMS and that is the source of truth for user account and account management of the user. User authentication on login occurs in the BYO UMS. Proxy users are created in the {{ site.product_short }} UMS upon authentication or subscription. When users are deleted from the UMS, proxy users are also deleted from {{ site.product_short }}.

    The synchronization for authentication and account management is accomplished using OIDC (Open ID Connect) protocol, an extension of OAuth 2.0. You must either provide a system with native OIDC support or update your UMS interface to support OIDC.

  • Purchase Notification - The web store is modified by you to send notifications of Commerce Events to the {{ site.product_short }} Entitlements API. The notification includes the USER ID and PRODUCT SKU. When {{ site.product_short }} receives the notification the appropriate update to the entitlement in the User record is recorded in the {{ site.product_short }} UMS. Thus bestowing or revoking the entitlement that enables the user to view content in {{ site.product_short }} apps. The technical Entitlements API interface is defined in {{ site.product_short }} Integration Details

Configuration details:

  • The subscription definitions in webstore and {{ site.product_short }} MUST be configured with the IDENTICAL SKU names.
  • The web store notifications are sent to a web hook URL that is presented from the {{ site.product_short }} CMS.

User Management Systems

The typical use case for {{ site.product_short }} stores is the integration of {{ site.product_short }} with a pre-existing, established web store. Less common is the launch of a new service with a new web store for managing commerce.

Externally Connected UMS

Pre-existing web stores are always integrated with user management and payment processing systems. In this case, your goal is to integrate {{ site.product_short }} within the pre-existing commerce configuration. Your UMS will remain the point of authentication and user record management. Therefore, the integration will use the customers UMS to authenticate and Authorize {{ site.product_short }} users. The user’s email address is used as the account ID and the entitlements are associated with that account. All user personal information data is stored in the BYO UMS.

External BYO UMS systems interface with {{ site.product_short }} using the OIDC interface protocol. Open ID Connect is an extension to OAuth that standardizes identity token schemas.

Mobile and Web UMS Configuration

  1. User selects to login in {{ site.product_short }} App
  2. App interacts with the OIDC provider to guide users through login flow. On success, access, refresh and identity tokens are provided to the app
  3. App uses access token to connect to {{ site.product_short }} backend
  4. {{ site.product_short }} backend validates access token

Connected and SmartTV UMS Configuration

  1. User selects to login in {{ site.product_short }} App
  2. App requests a device code
  3. User goes to Mobile or web device, enters authentication URL and device code
  4. Device interacts with OIDC provider to authenticate user
  5. Connected or Smart TV continuously poll the OIDC provider to find out if user authenticated, when authenticated, access, refresh and identity tokens are provided to the app
  6. App uses access token to connect to {{ site.product_short }} backend
  7. {{ site.product_short }} backend validates access token

{{ site.product_short }} User Management Autosync

When integrating an existing UMS with {{ site.product_short }}, proxy users must be created in the {{ site.product_short }} UMS so that entitlements created by {{ site.product_short }} Store sales notifications can be assigned to the proper user.

Users are maintained in {{ site.product_short }} as a result of events in the customer’s UMS. Through the OIDC interface, users are created in {{ site.product_short }} when they authenticate in the external UMS or if an entitlement is sent to {{ site.product_short }}. No manual synchronization is required.
